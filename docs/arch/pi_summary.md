# pi Agent System — Implementation Summary

Source: `/home/daniel/Projects/agents/pi`

---

## Overview

pi is a TypeScript-based coding agent system organized as an npm monorepo. It ships as a collection of layered packages, each independently publishable on npm under the `@mariozechner/` scope. The system spans a multi-provider LLM abstraction, a general-purpose agent runtime, a coding-focused agent with file and shell tools, a custom TUI framework with differential rendering, a Lit-based web UI component library, a Slack bot integration, and a GPU pod management CLI.

All packages share a lockstep version (currently 0.66.1) and are built with TypeScript (tsgo/tsc) with Biome for linting and formatting.

## Package: `@mariozechner/pi-ai`

The LLM abstraction layer. All provider-specific logic is encapsulated here; higher-level packages never call provider APIs directly.

### Type system

**`Model<TApi>`** — describes a model with its provider, API type, context window, max tokens, cost structure, supported input modalities, and whether it supports reasoning.

**`Api`** — a string union of known API wire formats:
- `openai-completions` — OpenAI Chat Completions API
- `openai-responses` — OpenAI Responses API
- `openai-codex-responses` — Codex-specific Responses API variant
- `azure-openai-responses` — Azure OpenAI Responses API
- `anthropic-messages` — Anthropic Messages API
- `bedrock-converse-stream` — AWS Bedrock Converse Stream
- `google-generative-ai` — Google Gemini API
- `google-gemini-cli` — Google Cloud Code Assist
- `google-vertex` — Google Vertex AI
- `mistral-conversations` — Mistral Conversations API

Custom string API types are supported via `(string & {})` intersection for extensibility.

**`Provider`** — a string union of known provider names (anthropic, openai, google, github-copilot, xai, groq, cerebras, mistral, openrouter, etc.) plus custom strings.

### Provider registry

Providers are registered lazily in `register-builtins.ts`. Each provider module exports:
- `stream<Provider>()` — raw provider-specific streaming
- `streamSimple<Provider>()` — unified streaming with reasoning/thinking support

Provider modules are loaded on first use via dynamic imports behind typed wrapper interfaces. The registry (`api-registry.ts`) maps `Api` strings to `{ stream, streamSimple }` pairs.

### Streaming API

**`stream(model, context, options)`** — returns an `AssistantMessageEventStream` for the given model's native API. Options are provider-specific.

**`streamSimple(model, context, options)`** — unified interface that accepts `SimpleStreamOptions` including a `reasoning` level (`minimal` | `low` | `medium` | `high` | `xhigh`) and maps it to provider-specific thinking parameters.

**`completeSimple(model, context, options)`** — convenience wrapper that awaits `streamSimple().result()`.

**`AssistantMessageEventStream`** extends `EventStream` and is an async iterable of `AssistantMessageEvent` values:
- `start` — stream opened, initial partial message
- `text_start` / `text_delta` / `text_end` — text content streaming
- `thinking_start` / `thinking_delta` / `thinking_end` — reasoning content
- `toolcall_start` / `toolcall_delta` / `toolcall_end` — tool call streaming
- `done` / `error` — terminal events

Calling `.result()` awaits the final `AssistantMessage` with `stopReason` (`stop`, `toolUse`, `error`, `aborted`), full `content` array, and `usage` (input, output, cacheRead, cacheWrite, cost).

### Model database

`models.generated.ts` is auto-generated by `scripts/generate-models.ts`, which fetches model catalogs from provider APIs and produces a static TypeScript map. `getModels(provider)` returns all known models for a provider; `getModel(provider, id)` finds a specific one.

### OAuth

Five built-in OAuth providers (`utils/oauth/`):

| Provider | Flow | Local callback port |
|---|---|---|
| Anthropic (Claude Pro/Max) | Authorization code + PKCE | `127.0.0.1:53692` |
| GitHub Copilot | Device code (no callback) | — |
| Google Cloud Code Assist (Gemini CLI) | Authorization code + PKCE | `127.0.0.1:8085` |
| Antigravity (Google multi-model) | Authorization code + PKCE | `127.0.0.1:51121` |
| OpenAI Codex (ChatGPT Plus/Pro) | Authorization code + PKCE | `127.0.0.1:1455` |

All providers implement `OAuthProviderInterface` with `login()`, `refreshToken()`, `getApiKey()`, and optional `modifyModels()`. The registry supports custom provider registration.

### Message transformation

`transform-messages.ts` handles cross-provider message format normalization. Each provider's stream function converts the unified `Message[]` into its native format and maps the response events back into the standardized `AssistantMessageEvent` sequence.

### Environment API keys

`env-api-keys.ts` maps provider names to environment variables (e.g. `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GEMINI_API_KEY`). Anthropic supports `ANTHROPIC_OAUTH_TOKEN` with higher precedence. AWS Bedrock and Google Vertex support credential-file detection.

## Package: `@mariozechner/pi-agent-core`

The generic, LLM-agnostic agent runtime. This package has no knowledge of files, shells, or coding — it only knows about messages, tools, and turn loops.

### Agent loop

`agent-loop.ts` implements the core loop:

1. **Prompt** — inject user messages into context, emit `agent_start` and `turn_start`.
2. **Stream** — call `streamSimple` with `convertToLlm(messages)`, emit `message_start` / `message_update` / `message_end`.
3. **Tool execution** — extract `toolCall` content blocks, validate arguments against TypeScript/TypeBox schemas, execute tools (sequential or parallel), emit `tool_execution_start` / `tool_execution_update` / `tool_execution_end`.
4. **Loop** — if the assistant requested tool calls, add results to context and return to step 2.
5. **Steering** — after each turn, check `getSteeringMessages()` for mid-run injections.
6. **Follow-up** — when the agent would stop, check `getFollowUpMessages()` for queued messages.
7. **End** — emit `agent_end` with all new messages.

**`agentLoop(prompts, context, config)`** returns an `EventStream<AgentEvent>` — the caller iterates events asynchronously while the loop runs.

**`agentLoopContinue(context, config)`** resumes from the current context without adding a new prompt (used for retries after compaction or error recovery).

### `AgentLoopConfig`

Configures the loop:
- `model` — the `Model` to use.
- `convertToLlm` — transforms `AgentMessage[]` → `Message[]` at the LLM call boundary.
- `transformContext` — optional pre-processing (context pruning, injection).
- `getApiKey` — dynamic key resolver for expiring OAuth tokens.
- `getSteeringMessages` / `getFollowUpMessages` — mid-run and post-run message injection.
- `toolExecution` — `"sequential"` or `"parallel"` (default parallel).
- `beforeToolCall` / `afterToolCall` — hooks that can block or override tool results.

### `AgentMessage`

A union of standard LLM `Message` types (user, assistant, toolResult) and extensible `CustomAgentMessages` (declaration-merging interface). This allows the coding agent to add custom message types (bash execution, compaction summary, branch summary) without modifying the core loop.

### `AgentTool`

Extends the base `Tool` schema with:
- `label` — human-readable name for UI.
- `prepareArguments` — optional shim for legacy argument formats.
- `execute(toolCallId, params, signal, onUpdate)` — runs the tool, returns `AgentToolResult<T>` with `content` and `details`.

### `Agent` class

Wraps the loop with:
- State management (`AgentState` — system prompt, model, thinking level, tools, messages).
- Event subscription with `subscribe(type, handler)`.
- `prompt(messages)` and `abort()`.
- Steering mode (`"all"` or `"one-at-a-time"`) for controlling mid-run message delivery.

## Package: `@mariozechner/pi-coding-agent`

The main product. Implements a coding agent with file manipulation tools, shell execution, session persistence, an extension system, and three run modes (interactive TUI, print, RPC).

### Run modes

**Interactive (TUI)** — `interactive-mode.ts` drives the TUI framework. Renders the chat transcript, handles user input, shows tool execution results with syntax highlighting and diffs, and manages overlays for model selection, settings, session browsing, and more.

**Print** — `print-mode.ts` runs a single prompt and prints the result to stdout. Used for scripted/CI usage.

**RPC** — `rpc-mode.ts` communicates via JSON lines over stdin/stdout. Commands include `prompt`, `steer`, `follow_up`, `abort`, `set_model`, `compact`, `bash`, `fork`, `get_state`, `export_html`, and many more. Events mirror the agent lifecycle. This mode enables programmatic control from external processes and editors.

### Tools

Seven built-in tools, each with pluggable operations for remote execution:

| Tool | Description |
|---|---|
| `read` | Read file contents. Supports text files and images (auto-resized). Truncation at 2000 lines or 50 KB. Supports offset/limit for ranged reads. |
| `write` | Create or overwrite files. Auto-creates parent directories. |
| `edit` | Exact text replacement. Takes an array of `{oldText, newText}` edits matched against the original file. Non-overlapping, non-incremental. Generates a unified diff. |
| `bash` | Execute shell commands via the user's shell. Streams output to a temp file. Truncation at 2000 lines or 50 KB (tail). Configurable timeout. Process tree cleanup on abort. |
| `grep` | Search file contents with regex support. Respects `.gitignore`. |
| `find` | Find files by name pattern. Respects `.gitignore`. |
| `ls` | List directory contents. |

Each tool follows the `ToolDefinition` interface with:
- A TypeBox schema for parameter validation.
- An `execute` function returning `AgentToolResult<TDetails>`.
- A `renderCall` function for TUI display of the tool invocation.
- A `renderResult` function for TUI display of the tool output (with syntax highlighting, diffs, expandable sections).

**File mutation queue** (`file-mutation-queue.ts`) — serializes concurrent `edit` and `write` calls to the same file path to prevent race conditions during parallel tool execution.

### System prompt

`system-prompt.ts` builds the prompt in layers:

1. **Role and tools** — identity, tool list with one-line snippets.
2. **Guidelines** — deduped bullets from tools and extensions.
3. **Pi documentation references** — paths to docs and examples.
4. **Append section** — from `APPEND_SYSTEM.md` or SDK option.
5. **Project context** — `AGENTS.md` / `CLAUDE.md` files walked from CWD to root, plus global from `~/.pi/`.
6. **Skills** — inline `SKILL.md` content.
7. **Date and CWD** — appended last.

A `SYSTEM.md` file (project or global) replaces the entire default body while keeping items 4–7.

Extensions can modify the prompt per-turn via the `before_agent_start` event.

### Session management

Sessions are stored as **append-only JSONL files** in `~/.pi/sessions/`. Each line is a typed entry:

| Entry type | Description |
|---|---|
| `session` | Header (version, ID, CWD, timestamp) |
| `message` | An `AgentMessage` (user, assistant, toolResult) |
| `thinking_level_change` | Thinking level was changed |
| `model_change` | Model was switched |
| `compaction` | Context compaction summary replacing older messages |
| `branch_summary` | Summary of an abandoned branch |
| `custom_message` | Extension-defined message type |
| `custom` | Extension-defined non-message entry |
| `label` | Named bookmark on an entry |
| `session_info` | Metadata update (title, model, etc.) |

The session is a **tree**, not a linear list. Each entry has `id` (UUID) and `parentId`. Forking creates a new branch from any point. Navigation between branches triggers branch summarization (LLM-generated summary of the abandoned path).

`SessionManager` provides:
- `newSession()` / `openSession()` / `forkSession()`
- `getBranch(leafId)` — path from root to leaf
- `addEntries(entries)` — append to the current branch
- `listSessions()` — scan all session files with metadata
- Session search by content

### Context compaction

Compaction triggers when estimated context tokens exceed `contextWindow − reserveTokens` (default 16,384).

**Algorithm:**
1. Walk entries backwards, accumulating token estimates (`chars / 4` heuristic, images at 1,200 tokens).
2. Stop at `keepRecentTokens` (default 20,000). This is the cut point.
3. Summarize discarded messages via the current model. The prompt asks for structured output: Goal, Constraints, Progress (Done/In Progress/Blocked), Key Decisions, Next Steps, Critical Context.
4. If a previous summary exists, the LLM merges rather than creates from scratch (iterative update).
5. Append file tracking: `<read-files>` and `<modified-files>` XML sections accumulated across compaction cycles.
6. Replace discarded entries with a single `compaction` entry.

**Split-turn handling:** If the cut falls inside a turn, the prefix is summarized separately with a shorter prompt, and the suffix is kept verbatim.

**Overflow recovery:** If the LLM returns a context-overflow error, the error message is removed from live context, compaction runs, and the turn is retried once.

### Extension system

Extensions are TypeScript modules loaded from `~/.pi/extensions/`, project `.pi/extensions/`, or CLI flags.

**Extension API** (`ExtensionAPI`):
- `registerTool(def)` — add an LLM-callable tool with schema, execute, and render functions.
- `registerCommand(name, handler)` — add a slash command.
- `registerShortcut(keybinding, handler)` — add a keyboard shortcut.
- `registerFlag(name, options)` — add a CLI flag.
- `on(event, handler)` — subscribe to lifecycle events.
- UI primitives: `notify()`, `select()`, `input()`, `confirm()`, `setStatus()`, `showOverlay()`.
- Agent control: `setModel()`, `setThinkingLevel()`, `setActiveTools()`, `sendUserMessage()`, `appendEntry()`.

**Extension events:**
- `before_agent_start` — modify system prompt, inject custom messages.
- `session_start` — one-time initialization.
- `session_compact` / `session_before_compact` — compaction lifecycle.
- `session_before_tree` / `session_before_fork` / `session_before_switch` — tree navigation.
- `input` — intercept user input before processing.
- `context` — transform context before LLM call.
- `tool_call` / `tool_result` — intercept tool execution.
- `message_start` / `message_update` / `message_end` — message lifecycle.
- `turn_start` / `turn_end` — turn lifecycle.
- `resources_discover` — inject custom skills/prompts/themes.
- `before_provider_request` — modify LLM request payload.
- `user_bash` — intercept bash commands.

**Extension isolation:** Each extension runs in the same Node.js process but gets its own `ExtensionAPI` proxy. Errors are caught per-extension and reported without crashing the session.

### Model registry

`ModelRegistry` manages:
- Built-in models from `pi-ai`'s generated catalog.
- Custom models from `~/.pi/models.json` (OpenAI-compatible and OpenAI-responses-compatible endpoints).
- Auth resolution: for each model, resolves API key from runtime override → `auth.json` → OAuth token (auto-refreshed) → environment variable → custom resolver.
- Available model filtering: only models with configured auth appear in model pickers.

### Auth storage

`AuthStorage` manages `~/.pi/auth.json` (mode 0600) with file locking for concurrent access:
- API key credentials: `{ type: "api_key", key: "..." }`
- OAuth credentials: `{ type: "oauth", access, refresh, expires, ... }`

OAuth tokens are refreshed automatically with `proper-lockfile` preventing race conditions between multiple pi instances.

Priority chain for key resolution:
1. Runtime override (`--api-key`)
2. API key in `auth.json`
3. OAuth token in `auth.json` (auto-refreshed)
4. Environment variable
5. Fallback resolver (custom providers from `models.json`)

### Settings

`SettingsManager` persists settings to `.pi/settings.jsonl` (project-local):
- Compaction settings (enabled, reserveTokens, keepRecentTokens)
- Output guard settings
- Thinking budgets
- Model scoping

### Resource loader

`ResourceLoader` discovers and loads:
- Extensions from `~/.pi/extensions/` and `.pi/extensions/`
- Skills from `~/.pi/skills/` and `.pi/skills/`
- Prompt templates from `~/.pi/prompts/` and `.pi/prompts/`
- Themes from `~/.pi/themes/` and `.pi/themes/`
- Context files (`AGENTS.md` / `CLAUDE.md`) walked from CWD upward
- System prompt overrides (`SYSTEM.md`, `APPEND_SYSTEM.md`)

### Skills

`SKILL.md` files with YAML frontmatter (`name`, `description`, optional `glob`/`when` conditions). Discovered from skill directories and rendered into the system prompt. Skills can reference other files in their directory via relative paths.

### Slash commands

Built-in commands: `/model`, `/thinking`, `/compact`, `/clear`, `/session`, `/fork`, `/tree`, `/diff`, `/tools`, `/settings`, `/login`, `/logout`, `/export`, `/share`, `/help`, and more. Extensions can register additional commands.

### Prompt templates

Markdown files in prompt directories. Invoked with `@template-name` in user input. Support frontmatter with `description` and `arguments`. Arguments are prompted interactively or supplied inline.

## Package: `@mariozechner/pi-tui`

Custom terminal UI framework with differential rendering. Not a wrapper around any existing TUI library.

### Architecture

**`TUI`** — the main runtime. Manages a component tree, keyboard input, focus, overlays, and rendering.

**Differential rendering** — on each frame, the TUI renders all components to line arrays, diffs against the previous frame, and emits only the ANSI escape sequences needed to update changed lines. This minimizes flicker and I/O.

**Component model** — every UI element implements `Component`:
- `render(width: number): string[]` — returns ANSI-formatted lines.
- `handleInput?(data: string)` — receives keyboard input when focused.
- `invalidate()` — clears cached render state.

**Focusable** — components with `focused: boolean` emit a `CURSOR_MARKER` (APC escape sequence) at the cursor position. TUI strips this marker and positions the hardware cursor there for IME support.

### Components

| Component | Description |
|---|---|
| `Editor` | Multi-line text editor with undo/redo, kill ring, selection, clipboard, word navigation, and configurable keybindings. Supports Kitty keyboard protocol. |
| `Markdown` | Streaming markdown renderer with syntax highlighting (via `shiki`), code block rendering, and inline formatting. |
| `SelectList` | Scrollable picker with fuzzy search, multi-column layout, truncation, and keyboard navigation. |
| `SettingsList` | Settings editor with toggle, enum, and input field types. |
| `Image` | Terminal image rendering via Kitty graphics protocol and iTerm2 inline images. Auto-detects terminal capabilities. |
| `Input` | Single-line text input with placeholder and cursor. |
| `Box` | Container with optional border, title, padding. |
| `Text` | Single-line styled text. |
| `TruncatedText` | Text with ellipsis truncation. |
| `Loader` | Animated spinner with label. |
| `CancellableLoader` | Loader with abort callback. |
| `Spacer` | Empty vertical space. |

### Overlays

`showOverlay(component, options)` displays a component as a floating overlay with configurable anchor, margin, size, and border. Overlays stack and capture input. Used for model pickers, settings, session browsers, login dialogs, etc.

### Keyboard handling

- **Kitty keyboard protocol** detection and parsing (`keys.ts`).
- **Key matching** via `matchesKey(data, keyId)` against string key descriptors (e.g. `"ctrl+o"`, `"shift+enter"`).
- **Keybindings** system (`keybindings.ts`) with configurable bindings stored in `~/.pi/keybindings.json`. All keybindings are configurable; hardcoded key checks are prohibited by project rules.
- **StdinBuffer** — batched input reader that splits pasted multi-key sequences from individual keypresses.

### Terminal capabilities

Auto-detected: Kitty keyboard protocol, Kitty graphics protocol, iTerm2 inline images, sixel, cell dimensions. Image rendering uses the best available protocol.

### Fuzzy matching

`fuzzy.ts` implements a fuzzy string matcher used by `SelectList` for search filtering and by autocomplete providers.

## `AgentSession` — the glue

`AgentSession` in the coding-agent package is the central orchestrator shared across all run modes. It connects:

- The `Agent` (from `pi-agent-core`) with its loop config.
- The `SessionManager` for persistence.
- The `ExtensionRunner` for extension lifecycle.
- The `ModelRegistry` for model and auth resolution.
- The `SettingsManager` for compaction and output settings.
- The `ResourceLoader` for skills, prompts, themes, and context files.

Key responsibilities:
- **Prompt submission** — validates model/auth, checks for pre-prompt compaction, fires `before_agent_start` extensions, calls `agent.prompt()`.
- **Event routing** — subscribes to `Agent` events and persists entries, fires extension events, handles auto-retry on retryable errors, triggers auto-compaction.
- **Auto-compaction** — after each turn, checks if context exceeds threshold or overflow occurred. Runs compaction with the current model.
- **Auto-retry** — retries on 529/overloaded, 429/rate-limit, and 5xx errors with exponential backoff. Configurable max attempts.
- **Session tree operations** — fork, switch, navigate branches. Branch summarization on navigation.
- **Tool management** — registers built-in and extension tools, rebuilds system prompt when tool set changes.
- **Model switching** — validates new model, updates system prompt, updates auth.
- **Bash execution** — direct shell execution outside the agent loop for user `/bash` commands.
- **HTML export** — renders session transcript to HTML with custom tool renderers.

## Event flow

A typical prompt-response cycle:

```
User types message
  ↓
InteractiveMode → AgentSession.prompt()
  ↓
before_agent_start extension event (may modify systemPrompt)
  ↓
Agent.prompt(messages)
  ↓
agent_start → turn_start
  ↓
convertToLlm(agentMessages) → Message[]
  ↓
streamSimple(model, context, options) → AssistantMessageEventStream
  ↓
message_start → message_update (streaming) → message_end
  ↓
For each toolCall in assistant response:
  tool_execution_start → execute tool → tool_execution_update → tool_execution_end
  ↓
  Tool results added to context
  ↓
  Next turn (loop back to streamSimple)
  ↓
turn_end → check getSteeringMessages() → check getFollowUpMessages()
  ↓
agent_end
  ↓
AgentSession: persist entries, check auto-compaction, check auto-retry
  ↓
InteractiveMode: update TUI
```

## Build and distribution

- **Language**: TypeScript, targeting ESM (`NodeNext` module resolution).
- **Monorepo**: npm workspaces with lockstep versioning.
- **Type checking**: `tsgo --noEmit` (the Go-based TypeScript type checker).
- **Linting**: Biome (`biome check --write --error-on-warnings`).
- **Testing**: Vitest. The coding agent test suite uses a faux LLM provider that returns scripted responses.
- **Build order**: tui → ai → agent → coding-agent → mom → web-ui → pods.
- **Release**: `npm run release:patch` / `release:minor` — bumps all packages, finalizes changelogs, commits, tags, publishes.
- **npm packages**: All published under `@mariozechner/` scope with `--access public`.
